# 束縛キャッシュ設計前提 (M2-6) — 2026-07-12設計レビュー決定(d)

## 問題

アンカー束縛は「provides台帳+History前送り」で**コンパイル時に**構築される
(FaceHandleはセッション内のメモリハンドル)。`.brep` キャッシュ(M1-1以降)に
ヒットしてコンパイルをスキップすると、**Historyが存在せず束縛が再構築できない**。

## 決定

**束縛表(anchor → 結果B-rep内の部分形状インデックス)を `.brep` と同一の
キャッシュキーで併存保存する。**

- キャッシュキー: hash(Part定義正規形 + ADCバージョン)(ADR-003 §6と同系)
- 束縛表は**決定的コンパイルの派生成果物**であり、正典に幾何IDを持ち込む
  わけではない(ADR-001に非抵触)。キー不一致=再コンパイルで束縛表も必ず
  作り直されるため、「黙って壊れる」経路にはならない
- 部分形状インデックスは「同一の.brepファイルをロードした際の決定的な
  部分形状列挙順」に対してのみ有効なローカル整数。キャッシュペア
  (.brep+束縛表)の外では無意味であり、設計記述・diff・エクスポートには
  一切現れない

## 実装時期

M2-6(結果キャッシュ)で実装する。M1では実装しない(コンパイルは常に
フル実行され、束縛はHistoryから毎回構築される)。

## 検証観点(M2-6受入に含めること)

1. キャッシュヒット経路とフルコンパイル経路で、全アンカーの束縛先が
   幾何的に一致すること(往復テスト)
2. Part定義変更でキーが変わり、古い束縛表が使われないこと
3. .brepのみ存在し束縛表が欠けた場合はキャッシュミス扱い(再コンパイル)
