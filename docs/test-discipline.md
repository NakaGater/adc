# テスト規律 (v0.1)

本プロジェクト自体のTDD(04-units-of-work.md 実装上の全体規律)における、
受入テストの書き方の規則。マイルストーンレビューで確認された判断を一般化して収める。

## 1. しきい値等号判定を受入テストに置かない(2026-07-12確定)

受入フィクスチャの実測値がチェッカーのしきい値と**ちょうど等しくなる**構成を避ける。
浮動小数演算の丸め(例: 半径ギャップ 27.5−26.5 が 0.99999…)で Pass/Fail が
反転するナイフエッジになるため。

- フィクスチャは**しきい値の明確に内側/外側**の値で構成する
  (M3の例: クリアランス受入をギャップちょうど1.0のφ53ではなく、1.5のφ52で固定)
- 等号ちょうどの挙動(≥ か > か)自体を検証したい場合は、幾何を経由しない
  単体テスト(チェッカーの判定関数に直接値を与える)で行う

## 2. 決定性はバイト同一で固定する

「同一入力→同一出力」(ADR-003)の受入は、フィールド個別比較ではなく
**正準出力(results.jsonl等)のバイト同一**で assert する。
宣言順入替・キャッシュ有無・実行回数の各軸で同一性を検証する
(M2-1決定性、M2-6 `--no-cache`、M3宣言順非依存が実例)。

## 3. ゴールデン値は実行環境と同一構成で生成する

数値ゴールデン(体積・面数・STEP)は**devcontainer内で生成した値**を正とする
(OCCTバージョン差による数値ズレの排除 — 06-deployment.md、M1-6)。
ホスト環境で生成した値をコミットしない。

## 4. エラー系はテーブル駆動で列挙する

E-SCHEMA-*等の静的検証は {名前, 入力, 期待エラーコード列} のテーブル駆動で書き、
ケース追加をテーブル1行の追加にする(M0-2が実例)。

## 5. E2E不能な性質は代替経路+正直な申告

語彙・環境の制約で受入E2Eが構成できない性質は、
(a) 単体テスト等の代替経路で数学・性質を検証し、
(b) マイルストーン報告で**E2E未達である旨を明示**し、
(c) 回収先ユニットをバックログ(04-units-of-work.md)に明記する。
黙って受入を弱めない(M1-3フィレット分割の△報告、M3 Angle mateが実例)。
