# Intent Statement

## 解決する問題

従来CADは3つの構造的欠陥を持つ:

1. **正典がジオメトリである。** フィーチャーツリーは人間の操作履歴であり、「なぜこの寸法なのか」という設計知識は保存されない。知識は図面の注記、ECOの文面、担当者の頭の中に散逸する
2. **AIが安全に触れない。** 幾何ID(Face#42)への参照は再生成で黙って壊れる(topological naming problem)。LLMに操作履歴を編集させることは、保証のない確率的操作を保証のないデータ構造に適用することであり、実務では成立しない
3. **検証が事後的・手動的である。** 干渉・肉厚・公差スタックのチェックは設計の後工程であり、CIのように「常に・自動で・変更のたびに」回る構造になっていない

## プロダクトビジョン

ADCは設計をソフトウェア開発と同型の構造に写像する:

| ソフトウェア | ADC |
|---|---|
| ソースコード | 設計記述 `design.ron`(要求+制約+根拠) |
| コンパイラ | ジオメトリコンパイラ(OCCT) |
| バイナリ | B-rep / STEP / メッシュ |
| テストスイート | アサーション+決定的チェッカー |
| CI | `adc check` on GitHub Actions |
| プルリクエスト | 制約差分+ジオメトリ差分+検証結果のレビュー |

これにより、AIエージェントは「夜間に設計空間を探索し、朝には検証済みの候補を提示する」存在になり、人間は「描く人」から「要求を定義しレビューする人」になる。

## ターゲットユーザー

- **一次:** 自動車部品の構想〜基本設計フェーズの機械設計者。ブラケット、ステー、ハウジング級の切削・板金部品を扱う
- **二次:** 設計エージェントを構築するAIエンジニア(MCP経由でADCを操作する)

## スコープ(MVP)

- 単品部品+小規模アセンブリ(2〜10部品)
- 製造プロセス: 切削(3軸前提)、板金
- 寸法公差+基本幾何公差(データム、位置度、平面度)、1D公差スタックアップ
- 検証: 干渉/クリアランス、質量特性、最小肉厚、板金設計規則、公差スタック(T1/T2チェッカー)
- STEP AP242出力、Git管理、CI実行
- 未確定パラメータ(`Open`)の区間絞り込み

## 非スコープ(明示的に作らない)

- クラスAサーフェス、意匠曲面(※パラダイム上は拡張可能: G2/G3連続性、曲率単調性、スキャン偏差など形式化された面品質基準はアサーション化できるため、「提案は確率的、判定は決定的」の構造が成立する。ただし専用のパッチ語彙とNURBS品質ツーリングを要する別エピック規模であり、MVP後に「ADC-Surface」として別途起票する)
- 大規模アセンブリ(>10部品)、PDM/PLM連携
- 本格FEM(質量特性・断面レベルの簡易チェックまで)
- 樹脂成形・金型設計(チェッカー語彙にも含めない)
- 5軸加工の工具経路検証
- GUI・3Dビューア(Phase 2。MVPはCLI+STEPを既存ビューアで確認)

## 成功基準

1. **往復完全性:** `design.ron` → build → STEP出力 → 定義変更 → 再build で、全アンカー参照が正しく再束縛される、または明示的なコンパイルエラーになる。黙った破壊がゼロであること
2. **決定性:** 同一入力に対し `adc check` の結果(margin含む)がバイト単位で再現すること
3. **実務者テスト:** 現役設計者が2〜10部品の実在ブラケットAssyをADCで記述し、「従来CADの構想設計より速い、または知識が残る点で優る」と評価すること
4. **エージェント適合性:** チェッカーのEvidence出力のみを材料に、LLMが違反を1〜3回の修正ループで解消できること(修復可能な粒度の検証)

## リスクと対応

| リスク | 対応 |
|---|---|
| opencascade-rsのAPI被覆不足 | 不足分は`cxx`ベースのFFI追加で対応。M1で被覆調査ユニットを最初に置く |
| 肉厚チェックの厳密解が重い | レイキャスト近似と明記し、密度パラメータで精度と速度をトレード |
| RON手書きの負担 | MVPはAIが書く前提。人間向け構文糖衣はPhase 2 |
| フィレット等OCCT操作の失敗 | 全フィーチャー操作の失敗を構造化エラー(E-FEATURE-FAIL)として返し、エージェントの修復ループに乗せる |
