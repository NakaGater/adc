# ユーザーワークフロー (v0.1)

MVPにおける標準的な利用手順。人間の窓口はClaude Code(対話)であり、ADCはMCP/CLI経由で呼ばれるツールである(ADR-006)。人間の操作は本質的に3種類——**要求を話す / 確認に答える / 差分と検証結果を見て承認する**——であり、RON文法やCLIコマンドの習得を前提としない。実装時はこの体験が成立するよう、エラーメッセージ・explain出力・report整形の品質を確認すること。

## パターンA: 新規部品の設計

1. **準備:** Git初期化済みリポジトリでClaude Codeを起動(ADCのMCPサーバーは `.mcp.json` で接続済み)
2. **要求を自然言語で伝える:** 「モーターマウントブラケット。6006ベアリング(外径φ55)を保持、フレームにM6×4本で締結、質量250g以下、材料A5052」
3. **エージェントが要求を正典に翻訳(Red):** ジオメトリより先にアサーション+rationaleを書く。曖昧な要求は `basis: Assumption` で明示し、未確定寸法は `Open(range)` で仮置きして人間に確認する
4. **生成→検証ループ(Green):** フィーチャー+アンカーを記述し `build_and_check`。FailはEvidence(アンカー参照+座標+実測値)を材料に自己修正。人間はループの中身を見ない
5. **人間の確認:** margin一覧の報告+`adc export --step` の出力を既存ビューアで目視。指摘は自然言語で返す(「ボス根元の肉が薄く見える」)
6. **コミット&PR:** CIで `adc check` が再実行され、PRに `adc report` のmargin表が付く。人間はレビューして承認。**rationale付きの設計が資産として残る**

## パターンB: 既存設計の変更

1. **変更を依頼:** 「ベアリングが6205(外径φ52)に変更。対応して」
2. **エージェントが影響範囲を先に調査:** いきなり編集せず `adc explain bore_d` で当該パラメータのrationale(旧根拠)と参照元(依存する制約・アサーション)を確認し、影響範囲を報告する。**過去の設計判断の「なぜ」が機械可読なため、引き継ぎ調査が会話1往復で終わる——パターンAとの最大の差分**
3. **パッチ→検証:** パラメータとrationaleを更新し `build_and_check`。キャッシュにより変更部品+関連Assyチェックのみ再計算。アンカーは意味参照のため嵌合先の参照は壊れない(壊れる場合はE-ANCHOR-BINDで顕在化)
4. **差分レビュー:** `adc diff` が制約差分(before→after+新rationale)、体積差、margin変化を提示。人間が見るのは形状全体ではなくこの差分
5. **承認:** PR上で確認して承認。変更履歴が「誰が・なぜ・何を根拠に」付きで蓄積され、次回のパターンBの材料になる

## 実装への含意(受入観点)

- `adc explain` はパターンB手順2の主役。参照元一覧(逆参照)の完全性が体験を左右する(M0-4)
- Evidence・構造化エラーは「エージェントが人間に頼らず1〜3回で直せる」情報量が受入基準(Intent成功基準4、US-16)
- `adc report` の出力はPRコメントとして人間が最初に見る顔。margin降順ソート・Fail先頭などの整形品質を軽視しない(M4-4)
