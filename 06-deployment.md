# デプロイ構成とデータ所在 (v0.1)

設計データ(正典・ビルド成果物・検証履歴)の機密性を前提としたデプロイ要件。`design.ron` は形状だけでなく設計知識(要求・rationale・Lesson・社内基準参照)を含むため、**従来のCADファイル以上の管理区分**で扱われることを想定する。

## 1. ADC本体の実装要件(全構成共通)

これらはM0から適用される実装上の制約であり、レビュー観点に含めること。

| 要件 | 内容 |
|---|---|
| **外部通信ゼロ** | adcバイナリはテレメトリ・自動更新・クラッシュレポート等の外部通信を一切行わない。ネットワークアクセスはCIで静的検知(依存クレート監査に`reqwest`等のHTTPクライアントが混入しないこと。ADR-006の依存グラフチェックと統合) |
| **オフライン完結** | build/check/diff/export はネットワーク不通環境(エアギャップ)で完全動作する |
| **データ生成の局所性** | キャッシュ・中間生成物はリポジトリ内(`.adc/`)またはユーザー指定ディレクトリのみに書く。ホームディレクトリへの暗黙の書き込みをしない |
| **秘密情報の非保持** | 認証情報・APIキーを一切保持しない(ADR-006) |

## 2. 構成パターン

### パターンS: 個人・OSS(標準)

- Git: GitHub / 任意
- CI: GitHub Actions(ホスト型ランナー)
- LLM: Claude Code(個人サブスク)
- データ所在: GitHub + Anthropic API(商用利用では既定でトレーニング不使用)

### パターンE1: 企業・クラウド統制型

- Git: GitHub Enterprise Cloud / EMU(IP allow list、SSO、監査ログ)
- CI: **セルフホストランナー**(check実行=ジオメトリ計算を社内で実行)
- LLM: **Amazon Bedrock**(`CLAUDE_CODE_USE_BEDROCK=1`)— 推論が自社AWSアカウント内で実行され、プロンプト(設計内容)がAWS境界内に留まる
- データ所在: 承認済みクラウド境界内

### パターンE2: 企業・完全オンプレ

- Git: GitHub Enterprise Server / GitLab Self-Managed / Gitea
- CI: オンプレランナー
- LLM: Bedrock(VPCエンドポイント経由)、または将来のオンプレ推論
- データ所在: 設計データは社外に一切出ない。ADCのオフライン完結要件(§1)によりエアギャップ環境でも成立(この場合LLM補助なしの手動編集+CLI運用)

## 3. LLM経路のデータフロー(明示)

| 経路 | 設計データの送信先 | 備考 |
|---|---|---|
| Claude Code(サブスク) | Anthropic API | 商用利用は既定でトレーニング不使用。許容可否は組織ポリシーに依存 |
| Bedrock | 自社AWSアカウント内の推論 | IAM統制・CloudTrail監査・コスト帰属。**企業標準** |
| LLMなし | 送信なし | ADCは決定的ツールとして単独で完全動作する |

導入ドキュメントでは、ユーザーが「何がどこに送られるか」を構成選択の前に理解できるよう、この表を必ず提示する。

## 4. リポジトリの管理区分に関する推奨

- `design.ron` を含むリポジトリは図面と同等以上の情報区分として扱う(rationale/Lessonは形状より漏洩影響が大きい)
- Lesson参照(`basis: Lesson`)の本文は正典に埋め込まず参照IDとし、本文はアクセス制御された知識基盤(Design Memory)側に置く——正典の可搬性と知識の集中管理を分離する
- CIアーティファクト(STEP、results.jsonl)の保持期間・公開範囲はリポジトリ本体と同一区分で設定する
