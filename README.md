# ADC — AI-native Declarative CAD

**正典は「要求+制約+根拠」。ジオメトリはビルド成果物。**

ADCは、AIエージェントを第一級の設計者として扱うために、ゼロから設計されたCADシステムである。従来CADが人間のマウス操作履歴を正典とするのに対し、ADCは宣言的な設計記述(型付きIR)を正典とし、ジオメトリカーネル(OCCT)をコンパイラとして扱う。設計のTDD——要求をアサーションとして先に書き(Red)、AIがジオメトリを生成して検証を通し(Green)、人間がレビュー・承認する——を骨格とする。

## コア原則

1. **正典の反転** — `.step` ではなく設計意図(`design.ron`)をコミットする。差分・マージ・レビューはGitのエコシステムに乗る
2. **意味的アンカー** — 幾何実体への参照は `bearing_bore` のような意味IDで行う。topological naming問題は「黙って壊れる」のではなく「コンパイルエラーとして顕在化」させる
3. **提案は確率的、判定は決定的** — LLMは幾何学的に間違える。全ての保証は決定的チェッカー(純関数)が与える
4. **未確定の第一級サポート** — 決まっていない寸法は `Open(range)` のまま保持し、検証ハーネスが実行可能区間を機械的に絞る
5. **rationaleはスキーマの一部** — 全ての制約に「誰が・何を根拠に」が付く。設計知識の復元(Design Memory)を後処理ではなくデータモデルで解決する
6. **ADCはLLM非搭載** — LLMは常にMCPクライアント側(Claude Code / Agent SDK)。Bedrockかサブスクかはユーザー環境の設定で切り替わり、ADCは関知しない(ADR-006)

## 技術スタック(確定事項)

- **言語:** Rust(edition 2021+)
- **カーネル:** OCCT 7.x — `opencascade-rs` クレートを土台に不足APIをFFIで追加
- **正典形式:** RON(Rust Object Notation)。独自構文フロントエンドはPhase 2以降
- **出力:** STEP AP242、results.jsonl(チェッカー結果)
- **CLI:** `adc build | check | diff | explain | export | report`(詳細は `07-cli.md`)

## 実装の進め方

`04-units-of-work.md` のマイルストーン順(M0→M6)に実装する。MVP=M0〜M5、Phase 2=M6。M0はジオメトリを一切含まない(スキーマ+パーサー+検証のみ)ため、OCCT環境構築と並行して着手できる。各ユニットは受入基準を満たすテストを先に書くこと(本プロジェクト自体もTDDで作る)。
